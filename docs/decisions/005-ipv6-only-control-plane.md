# ADR-005: IPv6-Only Control Plane with Local NAT IPv4

## Status
Accepted

## Date
2026-05-10

## Context
Mikrom previously managed IPv4 addresses globally using a centralized IPAM (IP Address Management) in the `mikrom-scheduler`. This introduced several issues:
- Tight coupling between the scheduler and the worker's local network state.
- Redundancy with the 6PN (IPv6 Private Network) mesh, which already provides deterministic identity.
- Database overhead and potential race conditions in IP allocation (`ip_allocations` table).
- Complexity in maintaining global state for transient local network details.

We needed a way to provide internet access to microVMs (which mostly requires IPv4) while simplifying the internal orchestration.

## Decision
Migrate to an IPv6-only control plane for all internal orchestration (scheduling, health checks, metrics, and routing) while offloading IPv4 management to the local agent nodes.

Key aspects of the decision:
1. **IPv6-Only Global State**: Remove all `ip_address` (IPv4) fields from database schemas (`deployments`, `jobs`, `workers`) and Protocol Buffer contracts.
2. **Deterministic 6PN Routing**: Use the deterministic IPv6 addresses generated by the 6PN mesh for all service-to-VM traffic.
3. **Local Hidden IPv4 NAT**: Each agent node manages its own local `10.0.0.0/8` pool (in-memory) to assign transient IPv4 addresses to microVMs. This range is used exclusively for outbound internet access via NAT on the host bridge (`mikrom-br0`).
4. **Strict IPv6 Health Checks**: Health checks performed by the agent must use the IPv6 address to ensure mesh connectivity is functional.

## Alternatives Considered

### Dual-Stack Global Management
- **Pros**: Full visibility of both IPv4 and IPv6.
- **Cons**: Doubled the complexity of IPAM logic and database storage. No real benefit for internal orchestration since 6PN is sufficient.

### IPv6-Only Internet Access (NAT64/DNS64)
- **Pros**: Cleanest architecture, no IPv4 at all.
- **Cons**: Requires nodes to have stable upstream IPv6 connectivity and complex NAT64 gateways. Many third-party container images and external services still lack robust IPv6 support.

## Consequences
- **Simplified Scheduler**: The `Ipam` module and `ip_allocations` table have been deleted. Scheduling is now faster and more reliable.
- **Decoupled Workers**: Workers no longer need to sync their local IPv4 pool with the global database.
- **Improved Observability**: `VmMetrics` and CLI status now explicitly show the 6PN IPv6 address, reinforcing the identity-based network model.
- **Health Check Rigor**: Failures in the 6PN mesh are now immediately detected by health checks.
- **Compatibility**: Outbound internet via IPv4 NAT remains fully functional for microVM workloads.
